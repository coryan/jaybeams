1 # JayBeams Style Guide and Coding Standards
2 
3 ## Introduction
4 
5 This document describes the coding standards used in JayBeams, and the
6 rationale behind them. If you disagree with the standard please open
7 a issue in github and/or send a PR to change this document before
8 changing the code.
9 
10 ## Naming Guidelines
11 
12 ### Identifiers
13 
14 Identifiers use lowercase characters, with words separated by
15 underscore characters. All global identifiers (except pre-processor
16 macros) must be in the `jb::` namespace.
17 
18 **Rationale:** this is the naming adopted in the C++ standard and
19  Boost. It used to be the case that uppercase names were "safer" for
20  the application, because in implementations lacking namespaces the
21  system or standard could introduce a conflicting name without
22  warning.
23 
24 ### Function-like Macros
25 
26 In general JayBeams tries to not use macros, but they are sometimes
27 the best (or only) way to implement a feature. Function-like macros
28 should be named like functions, using lowercase characters with words
29 separated by underscores.
30 
31 ### Object-like Macros
32 
33 The only object-like macros used in JayBeams are intended to set
34 compile-time configurable parameters.
35 Such macros are used in a single .cpp file, start with the path of the
36 file in ALL CAPS, followed by the name of the constant in all lowercase.
37 
38 For example, suppose a compile-time configurable parameter is called
39 `max_slots` and is used in `jb/foo_bar/baz.cpp`, the macro name shall
40 be `JB_FOO_BAR_BAZ_max_slots`.
41 
42 ### Namespaces
43 
44 Namespaces are also lowercase, with the path of the namespace matching
45 the path of the files. So, for example, all classes and functions in
46 the `jb::foo::bar` namespace must be defined in files contained by the
47 `jb/foo/bar` directory.
48 
49 **Rationale:** we want to make it easy to find classes and functions.
50 
51 #### Exception: the `jb::*::defaults` namespaces
52 
53 JayBeams defines a series of namespaces containing default values for
54 configuration parameters. The objects in these namespaces do not need
55 a sub-directory.
56 
57 **Rationale:** the namespace simply defines a number of constants,
58  typically used in a single `.cpp` file, creating a complete new
59  directory for them is overkill.
60 
61 ### Filenames
62 
63 Filenames are all lowercase, no file can be longer than 128 bytes in
64 length.
65 The file name should match the name of the main class or function
66 defined in the class.
67 
68 **Rationale:** This makes it easier to find the classes and
69  functions. Always using lowercases avoid conflicts in operating
70  systems that do not distinguish between uppercase and lowercase
71  characters in filenames. Some modern operating systems impose
72  limits on filename length [ref](https://en.wikipedia.org/wiki/Comparison_of_file_systems)
73 
74 ### No "util" Files
75 
76 **Rationale:** Util files become hard to maintain and distinguish from
77 each other, if you have standalone functions just create a separate
78 pair of files for them.
79 
80 ## Use Exceptions Sparingly
81 
82 Exceptions should represent, well, exceptional conditions. Normal
83 errors that we expect (e.g. missing files, closed sockets, etc) are
84 not exceptional conditions and the API should include treatment for
85 them.
86 
87 **Rationale:** this is the standard practice in C++, read "The Design
88  and Evolution of C++".
89 
90 ## External Depedendencies
91 
92 JayBeams will prefer to use an external dependency over reimplementing
93 significant functionality. Obviously there is a tradeoff for very
94 small external dependencies, developers should apply their best
95 judgement. As a guideline, external libraries that implement less
96 than 100 lines of code are probably too uninteresting to depend on.
97 
98 ### Preferred Libraries
99 
100 JayBeams will prefer to use functionality from the standard C++
101 library when available. If not available, JayBeams will prefer to use
102 libraries for [boost](http://www.boost.org). If neither the C++
103 standard nor Boost provide the desired functionality then the
104 developers should describe the dependency and document the rationale
105 for it in this document.
106 
107 #### FFTW Library
108 
109 Implement the Fast Fourier Transform (FFT) algorithm for modern CPUs,
110 taking advantage of vectorized instructions and many other
111 optimizations.
112 
113 **Rationale:** Largely grandfathered in. Has a good reputation as a
114 fast FFT library.
115 
116 #### YAML-CPP Library
117 
118 Implement a YAML [yaml.org](http://www.yaml.org) parser for C++.
119 
120 **Rationale:** Largely grandfathered in.
121 
122 #### Boost.Compute Library
123 
124 Boost wrappers for OpenCL.
125 
126 **Rationale:** accepted into boost.org as of 1.61.0, though not
127 included in many Linux distributions yet. Wanted better (safer, more
128 expressive) wrappers than what OpenCL provides natively.
129 
130 #### clFFT Library
131 
132 A FFT library for OpenCL implemented by AMD.
133 
134 **Rationale:** Largely grandfathered in.

136 ## Formatting and Whitespace
137 
138 JayBeams uses
139 [clang-format](http://clang.llvm.org/docs/ClangFormat.html) for
140 low-level formatting issues. We use version 3.8 or higher to do the
141 formatting. The configuration file is included with
142 the source code.
143 
144 Notice that this includes the order of `#``include` directives. We
145 follow the suggestions from Lakos: first include the main header file,
146 then local files, the increasingly lower dependencies. The objective
147 is to detect missing dependencies in the header files early.
148 With `clang-format` the main header file must be labeled specially by
149 using double quotes instead of angle brackets for the include
150 directive. That is, please use `#``include "foo/bar/baz.h"` instead of
151 `#``include <foo/bar/baz.h>`.
152 
153 **Rationale:** arguing about whitespace is not productive, let the
154 robots do that.
