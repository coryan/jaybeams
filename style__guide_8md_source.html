1 # JayBeams Style Guide and Coding Standards
2 
3 ## Introduction
4 
5 This document describes the coding standards used in JayBeams, and the
6 rationale behind them. If you disagree with the standard please open
7 a issue in github and/or send a PR to change this document before
8 changing the code.
9 
10 ## Header Files
11 
12 ### All headers should be self-contained.
13 
14 One should be able to `#``include` a header and compile it without
15 requiring any additional headers before or after.
16 
17 ### The `#`define Guard
18 
19 All headers should include a `#``ifndef/``#``define/``#``endif` guard.
20 The guard must be the first line in the header.
21 The name of the guard is constructed by the full path of the header
22 file from the root of JayBeams, replacing both slashes (`/`) and
23 period (`.`) characters by an underscore (`_`).
24 For example, the include guard for `foo/bar/baz_qux.hpp` becomes
25 `foo_bar_baz_qux_hpp`.
26 There must be no double underscores in a guard name, that limits your
27 options for filenames.
28 
29 **Example:** the `foo/bar/baz_qux.hpp` file **must** be protected by:
30 
31 ```c++
32 #ifndef foo_bar_baz_qux_hpp
33 #define foo_bar_baz_qux_hpp
34 // ... your code here
35 #endif // foo_bar_baz_qux_hpp
36 ```
37 
38 **Rationale:** protecting against multiple includes is a standard C++
39  practice, we will not elaborate further. Making that the first line
40  enables some compilers to optimize parsing of the header if it is
41  already included. Making the name of the guard algorithmic removes
42  a debate. Identifiers with double underscores are reserved for the
43  implementation in POSIX, so we avoid them.
44 
45 ### Forward Declarations
46 
47 Use them when needed or they make sense.
48 
49 ### Inline Functions
50 
51 Single line functions should be inlined, virtual functions, even if
52 single line, should not be inlined.
53 Inline function should not be separate from the class definition.
54 
55 **Use:**
56 
57 ```c++
58 class foo {
59 public:
60  inline int answer() const {
61  return 42;
62  }
63 ```
64 
65 **Do Not Use:**
66 
67 ```c++
68 class foo {
69 public:
70  int answer() const;
71 };
72 inline int Foo::answer() const {
73  return 42;
74 }
75 ```
76 
77 **Rationale:** We want to avoid function call overhead for trivial
78  functions, but inlining rarely works for virtual functions.
79  We want to minimize typing.
80 
81 ### Name of Includes
82 
83 The name of the include must match the name of the main class or
84 function in the include.
85 Define a single class per include, do not define many classes in the
86 same header.
87 Multiple function overloads can be included in the same header, but
88 avoid many unrelated functions in the same header.
89 
90 ### Order of Includes
91 
92 The main include for a source file goes first.
93 The includes in the same directory,
94 then includes in JayBeams,
95 then boost and other third-party libraries,
96 then C++ libraries.
97 
98 The order is enforced by clang-format, so we do not define it further,
99 the tool does it for us.
100 
101 **Rationale:** Lakos' argument, we want to include the header early to
102 detect missing dependencies as soon as possible.
103 
104 ### Namespaces
105 
106 Namespaces are also lowercase, with the path of the namespace matching
107 the path of the files. So, for example, all classes and functions in
108 the `jb::foo::bar` namespace must be defined in files contained by the
109 `jb/foo/bar` directory.
110 
111 **Rationale:** we want to make it easy to find classes and functions.
112 
113 #### Exception: the `jb::*::defaults` namespaces
114 
115 JayBeams defines a series of namespaces containing default values for
116 configuration parameters. The objects in these namespaces do not need
117 a sub-directory.
118 
119 **Rationale:** the namespace simply defines a number of constants,
120  typically used in a single `.cpp` file, creating a complete new
121  directory for them is overkill.

123 ## Naming Guidelines
124 
125 ### Identifiers
126 
127 Identifiers use lowercase characters, with words separated by
128 underscore characters. All global identifiers (except pre-processor
129 macros) must be in the `jb::` namespace.
130 
131 **Rationale:** this is the naming adopted in the C++ standard and
132  Boost. It used to be the case that uppercase names were "safer" for
133  the application, because in implementations lacking namespaces the
134  system or standard could introduce a conflicting name without
135  warning.
136 
137 ### Function-like Macros
138 
139 In general JayBeams tries to not use macros, but they are sometimes
140 the best (or only) way to implement a feature. Function-like macros
141 should be named like functions, using lowercase characters with words
142 separated by underscores.
143 
144 ### Object-like Macros
145 
146 The only object-like macros used in JayBeams are intended to set
147 compile-time configurable parameters.
148 Such macros are used in a single .cpp file, start with the path of the
149 file in ALL CAPS, followed by the name of the constant in all lowercase.
150 
151 For example, suppose a compile-time configurable parameter is called
152 `max_slots` and is used in `jb/foo_bar/baz.cpp`, the macro name shall
153 be `JB_FOO_BAR_BAZ_max_slots`.
154 
155 
156 ### Filenames
157 
158 Filenames are all lowercase, no file can be longer than 128 bytes in
159 length.
160 The file name should match the name of the main class or function
161 defined in the class.
162 
163 **Rationale:** This makes it easier to find the classes and
164  functions. Always using lowercases avoid conflicts in operating
165  systems that do not distinguish between uppercase and lowercase
166  characters in filenames. Some modern operating systems impose
167  limits on filename length [ref](https://en.wikipedia.org/wiki/Comparison_of_file_systems)
168 
169 ### No "util" Files
170 
171 **Rationale:** Util files become hard to maintain and distinguish from
172 each other, if you have standalone functions just create a separate
173 pair of files for them.
174 
175 ## Use Exceptions Sparingly
176 
177 Exceptions should represent, well, exceptional conditions. Normal
178 errors that we expect (e.g. missing files, closed sockets, etc) are
179 not exceptional conditions and the API should include treatment for
180 them.
181 
182 **Rationale:** this is the standard practice in C++, read "The Design
183  and Evolution of C++".
184 
185 ## Aliases
186 
187 Use aliases to increase clarity, specially in template code where type
188 traits are a common practice to improve readability and usability.
189 Prefer the C++11 `using foo = bar;` aliases over the `typedef bar
190 foo;` aliases.
191 
192 **Rationale:** The use of type aliases is a standard practice, we will
193  not elaborate further.
194  The use of `using` aliases vs. `typedef` aliases is for
195  consistency: `using` is required for template aliases, so we use it
196  everywhere.
197 
198 ## External Depedendencies
199 
200 JayBeams will prefer to use an external dependency over reimplementing
201 significant functionality. Obviously there is a tradeoff for very
202 small external dependencies, developers should apply their best
203 judgement. As a guideline, external libraries that implement less
204 than 100 lines of code are probably too uninteresting to depend on.
205 
206 ### Preferred Libraries
207 
208 JayBeams will prefer to use functionality from the standard C++
209 library when available. If not available, JayBeams will prefer to use
210 libraries for [boost](http://www.boost.org). If neither the C++
211 standard nor Boost provide the desired functionality then the
212 developers should describe the dependency and document the rationale
213 for it in this document.
214 
215 #### FFTW Library
216 
217 Implement the Fast Fourier Transform (FFT) algorithm for modern CPUs,
218 taking advantage of vectorized instructions and many other
219 optimizations.
220 
221 **Rationale:** Largely grandfathered in. Has a good reputation as a
222 fast FFT library.
223 
224 #### YAML-CPP Library
225 
226 Implement a [YAML](http://www.yaml.org) parser for C++.
227 
228 **Rationale:** Largely grandfathered in.
229 
230 #### Boost.Compute Library
231 
232 Boost wrappers for OpenCL.
233 
234 **Rationale:** accepted into boost.org as of 1.61.0, though not
235 included in many Linux distributions yet. Wanted better (safer, more
236 expressive) wrappers than what OpenCL provides natively.
237 
238 #### clFFT Library
239 
240 A FFT library for OpenCL implemented by AMD.
241 
242 **Rationale:** Largely grandfathered in.

244 ## Formatting and Whitespace
245 
246 JayBeams uses
247 [clang-format](http://clang.llvm.org/docs/ClangFormat.html) for
248 low-level formatting issues. We use version 3.8 or higher to do the
249 formatting. The configuration file is included with
250 the source code.
251 
252 Notice that the formatting prescribes the order of `#``include`
253 directives.
254 We follow the suggestions from Lakos: first include the main header file,
255 then local files, the increasingly lower dependencies. The objective
256 is to detect missing dependencies in the header files early.
257 With `clang-format` the main header file must be labeled specially by
258 using double quotes instead of angle brackets for the include
259 directive. That is, please use `#``include "foo/bar/baz.h"` instead of
260 `#``include <foo/bar/baz.h>`.
261 
262 **Rationale:** arguing about whitespace is not productive, let the
263 robots do that.
